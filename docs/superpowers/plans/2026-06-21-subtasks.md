# Subtasks ‚Äî Implementation Plan

Companion to `docs/superpowers/specs/2026-06-21-subtasks-design.md` ‚Äî read that first for the *why*. This is the task-by-task *how*, ordered backend-first since the frontend depends on the new field/commands.

Grounded in two research passes against the actual code (not assumed): Task/Project storage is file-based (one markdown file per task, one JSON array for all projects ‚Äî no DB, no migrations needed for a new field), `create_task` already walks the ancestor chain for defaults, `update_task` takes a full `Task` struct and unconditionally overwrites editable fields, `delete_task` does nothing beyond removing one file today, and the project cascade-delete pattern (`projects_to_delete` + descendant counting) is the model to mirror for task-side cascading.

## Backend (Rust, src-tauri/src/)

1. **`Task.subtask_project_id: Option<String>`** ‚Äî add to the struct in `task.rs` with `#[serde(default, skip_serializing_if = "Option::is_none")]` (same pattern as `series_id`), initialize `None` in `Task::new`. No new field on `Project` (see spec's "no back-pointer" decision). Test: markdown round-trip with and without the field set, and that old task files missing the field still deserialize.

2. **`ensure_subtask_container` command** ‚Äî new `#[tauri::command]` in `commands.rs`, signature `(state, parent_task_id: String) -> Result<Project, String>`. Logic as a testable helper `fn get_or_create_subtask_container(tasks: &mut Vec<Task>, projects: &mut Vec<Project>, parent_task_id: &str) -> Result<Project, String>`: find the parent task (error if missing); if `subtask_project_id` is already `Some`, look it up in `projects` and return it; otherwise build a new `Project` (name = parent task's current title, `parent_id` = parent task's `project_id`, default board/defaults, fresh id) ‚Äî **skip the sibling-uniqueness-name check** `create_project` does for user-facing creation (this container is hidden from any UI where name collision would be confusing; see spec), push it, set the parent task's `subtask_project_id`, and return the new project. Command body: load tasks+projects, call the helper, save both lists, return. Register in `lib.rs`'s `generate_handler!`. Tests: creates on first call, returns the same container on a second call without duplicating, errors on a missing task id.

3. **Rename/move-sync inside `update_task`** ‚Äî testable helper `fn synced_subtask_container(projects: &[Project], container_id: &str, task_title: &str, task_project_id: &str) -> Option<Project>` returning an updated copy (name + `parent_id` set to the task's current values) or `None` if the container can't be found. Call unconditionally (no need to detect *whether* title/project_id changed ‚Äî resyncing to current values is idempotent and simpler than threading old-value comparisons through `apply_task_update`'s call site) from `update_task`'s body whenever `existing.subtask_project_id` is `Some`, after `apply_task_update` runs: look up + replace the container in the loaded `projects` list, save. Tests: title change renames the container, project_id change re-parents it, a no-op update leaves it unchanged, no container present is a no-op.

4. **Cascade-delete inside `delete_task`** (parent-task-deletion case) ‚Äî testable helper `fn tasks_and_project_to_delete_with_subtasks(tasks: &[Task], projects: &[Project], task_id: &str) -> (Vec<Task>, Option<Project>)` mirroring `projects_to_delete`'s shape: the target task is *not* included in the returned `Vec<Task>` (that's deleted by the existing single-file-delete path) ‚Äî this returns the *subtasks* (tasks whose `project_id` equals the target's `subtask_project_id`) plus the container `Project` itself, both empty/`None` if the target has no container. Command body: compute this before deleting anything, delete each subtask file, remove the container from the projects list + save, *then* delete the target task's own file (existing behavior, unchanged). Tests: deleting a task with subtasks returns the right set, a task with no container returns empty/`None`, subtask files actually get removed from disk.

5. **Empty-container cleanup inside `delete_task`** (subtask-deletion case ‚Äî the task being deleted *is itself* a subtask in someone else's container) ‚Äî testable helper `fn owning_task_if_container_now_empty(tasks: &[Task], deleted_task_project_id: &str, deleted_task_id: &str) -> Option<&Task>`: find a task whose `subtask_project_id == deleted_task_project_id` (the owner); if found, check whether any *other* task (excluding the one being deleted) still has `project_id == deleted_task_project_id`; if none do, return the owner so the caller can clear its `subtask_project_id` and remove the container project. Command body: run this check (alongside #4's check ‚Äî a single `delete_task` call handles both "deleting a task that owns a container" and "deleting a task that's the last subtask in someone else's container," they're independent conditions, not mutually exclusive in code even though a given task can't realistically be both in this app's data model) before deleting the target file; if it finds an owner, clear that owner's `subtask_project_id`, save it, and remove the now-empty container from the projects list. Tests: deleting the last subtask clears the owner and removes the container, deleting a subtask when siblings remain does neither, deleting a normal (non-subtask) task is a no-op for this path.

6. Run `cargo fmt`, `cargo clippy --lib -- -D warnings`, `cargo test --lib` after each numbered step, not just at the end ‚Äî this mirrors how the Subprojects backend work actually went (caught real bugs early rather than after everything was written).

## Frontend (src/lib/)

7. **`Task` type** (`types.ts`) gains `subtask_project_id?: string`. **`api.ts`** gains `ensureSubtaskContainer(parentTaskId: string): Promise<Project>` wrapping the new command.

8. **`subtasks.ts`** (new file, pure functions, mirrors `projectTree.ts`'s style) ‚Äî
   - `isSubtask(task: Task, allTasks: Task[]): boolean` ‚Äî does some other task's `subtask_project_id` equal this task's `project_id`?
   - `containerOwner(projectId: string, tasks: Task[]): Task | undefined` ‚Äî reverse lookup, the "no marker field" mechanism from the spec.
   - `subtasksOf(parentTask: Task, allTasks: Task[]): Task[]`
   - `subtaskProgress(parentTask, allTasks, doneStatusId, cancelledStatusId): { done: number; total: number }` ‚Äî both counts exclude cancelled subtasks.
   - `allSubtasksDone(parentTask, allTasks, doneStatusId, cancelledStatusId): boolean` ‚Äî requires at least one non-cancelled subtask (vacuously "true" with zero non-cancelled subtasks would be misleading, not an actual completion).
   - `subtaskPrefillTitle(parentTask: Task, priorities, statuses): string` ‚Äî builds the leading quick-add token string (`#tag !priority due:... sch:... est ...`) described in the spec's pre-fill decision, trailing space included, empty string if the parent has nothing to carry over.
   Tests for every function above ‚Äî this file is the bulk of the feature's real logic and the easiest to get fully covered.

9. **NL parsing**: extract the quote-consuming token-pulling loop out of `tryResolveProjectPath` (`naturalLanguage.ts`) into a small shared primitive (e.g. `consumeQuotedPhrase(tokens, index): { text: string; consumed: number } | undefined`, pulling in tokens while a `"` is unbalanced, no `/`-splitting) used by both it and a new `trySubtaskParentName`. Add `subtaskParentName?: string` to `ParsedTaskInput`, parsed from `sub <phrase>` (quoted if multi-word, same convention as the project-path work). Tests mirror the `+Project` path ones: bare single-word name, quoted multi-word name, unterminated quote falls back to leaving `sub` untouched in the title, `sub` takes precedence when both `sub` and `+Project` appear.

10. **`AddTaskModal.svelte` wiring**: new derived `matchedParentTask` (resolves `parsed.subtaskParentName` against `tasksState.items`, filtered to active+non-subtask per the spec's autocomplete-scope decision even for literal typed resolution, not just autocomplete) taking precedence over `matchedProject` when present. When `matchedParentTask` is set, `preview.project` becomes `"Subtask of {title}"` (no real project id resolved yet ‚Äî see spec). `handleSubmit` awaits `ensureSubtaskContainer(matchedParentTask.id)` before calling `createTask` when this path is taken. New optional prop `initialTitle?: string` read by the existing reset effect instead of the hardcoded `""`, for the button-driven entry points (#11) to seed via `subtaskPrefillTitle`. New `sub` entry in the autocomplete dropdown's option set (task titles, filtered to active+non-subtask, quoted on insert when multi-word ‚Äî same insertion-quoting helper pattern as `projectPathSuggestions`, reused not reinvented).

11. **"Create Subtask" entry points** ‚Äî a shared handler (e.g. exported from `subtasks.ts` or a tiny new component-local helper) that opens `AddTaskModal` with `projectFilter` unset, `initialTitle = subtaskPrefillTitle(parentTask, ...)`, gated on `!isSubtask(task, tasksState.items)`:
    - `TaskCard.svelte`: a button after the existing `.task-meta` block.
    - `TaskEditDialog.svelte`: a button alongside the existing Save/Cancel/Delete row.
    - `WeekBarItem.svelte`: a new `onCreateSubtask` prop beside the existing `onEdit`, threaded from `WeekView.svelte` the same way `onEdit` already is.

12. **Glued nested rows in `TaskCard.svelte`** ‚Äî gated on `displayState.showSubtasks`, a `<ul>` of subtask rows after `.task-meta`: priority indicator left, title, a clickable status dot right that opens a small inline status picker (reuse whatever status-picker affordance already exists for the main board columns if one is componentized, otherwise a minimal new one ‚Äî check `StatusSettings.svelte`/board column headers for an existing reusable piece before writing a new one). Progress badge (`subtaskProgress`) shown on the parent card regardless of `showSubtasks` (the badge is a summary, not the expanded list).

13. **`displaySettings.svelte.ts`**: new `showSubtasks: boolean` following the exact `showPriorityChip` pattern (constant key, default, state field, `setShowSubtasks`, `initDisplay()` restore line). Wire a toggle into whichever settings component already exposes `showPriorityChip`'s control.

14. **Board/Week/Calendar filtering**: wherever tasks are currently filtered for display (need to locate the exact filtering point per view ‚Äî likely inside `KanbanBoard.svelte`/`WeekView.svelte`/the calendar view's data-prep), exclude tasks for which `isSubtask(task, allTasks)` is true *and* `displayState.showSubtasks` is false ‚Äî "vanish entirely" per the spec, not just un-nest.

15. **Sidebar exclusion**: wherever the project tree is built for the sidebar (`Sidebar.svelte`/`ProjectTreeNode.svelte`), filter out any project for which `containerOwner(project.id, tasksState.items)` is defined.

16. **Subtask container's own board page**: in `src/routes/projects/[id]/+page.svelte`, branch on `containerOwner(project.id, tasksState.items)` ‚Äî if defined, render `"Subtasks of {owner.title}"` in place of the existing `ancestorsOf`-driven breadcrumb; the rest of the page (the `KanbanBoard` itself) needs no change, since the container is a perfectly normal project as far as the board component is concerned.

17. **All-done popup**: a small reactive module (e.g. `subtaskCompletion.svelte.ts`) tracking, per parent-task-id, whether it was "all done" on the *previous* reactive run ‚Äî exposing a pure function `newlyAllDoneTaskIds(previouslyAllDone: Set<string>, tasks: Task[], doneStatusId, cancelledStatusId): { newlyDone: string[]; stillAllDone: Set<string> }` (testable in isolation) plus a thin `$effect`-driven wrapper that calls it and surfaces a dialog (`AllSubtasksDoneDialog.svelte`, new ‚Äî offers "Mark parent done" / "Delete subtask list" / dismiss) for each id in `newlyDone`. The "don't reshow for the same completion" rule falls out naturally from this being edge-triggered (compares against the previous run, not a persisted dismissal flag) ‚Äî no extra state needed beyond the running `Set`.

18. **Cascade-delete confirmation**: `TaskCard.svelte`/`TaskEditDialog.svelte`'s existing `handleDelete` (which already branches on `task.series_id`/`generalState.confirmTaskDeletion`) gains one more check: if the task owns a container (`subtask_project_id` set), compute the subtask count via `subtasksOf` and pass a richer message into the *existing* `ConfirmDialog` (no new dialog component) ‚Äî e.g. "This will also delete N subtasks."

19. Full verification: `cargo test`/`fmt`/`clippy`, `npx vitest run`, `npx svelte-check`, `npm run build`, then a manual Playwright pass against the mocked-IPC harness covering: creating a subtask from each of the three entry points, the glued nested view with a status-dot change, the progress badge, the show/hide setting actually hiding rows, the all-done popup firing once and not re-firing, cascade-delete's confirmation count, and the container board's overridden breadcrumb.

## Notes on sequencing

Steps 1‚Äì6 (backend) are a clean, fully-testable unit ‚Äî do them first and completely before touching the frontend, mirroring how the original Subprojects backend work went. Within the frontend, step 8 (`subtasks.ts`) is the dependency root for nearly everything else (10‚Äì18 all read from it) and should be done and fully tested before any UI wiring starts, to avoid discovering a wrong function signature after three components already call it.
